What is an API?

An Application Programming Interface (API) defines the rules for how applications can communicate with each other.

API Functions

  • Allows controlled data exchange
  • Enables service integration
  • Facilitates client-server communication

Common API Types

  • RESTful APIs
  • GraphQL APIs
  • SOAP APIs
  • WebSockets
  • gRPC

API Analogy: Restaurant

Think of an API like a restaurant system:

  • Menu = API documentation (available options)
  • Waiter = API (takes requests, returns responses)
  • Kitchen = Backend system (processes requests)
  • Order = API request (what you want)
  • Food served = API response (what you get back)

RESTful APIs

REST (Representational State Transfer) is an architectural style for designing networked applications.

Core Principles:

  • Stateless: Each request contains all information needed
  • Resource-Based: Everything is a resource identified by URL
  • Standard Methods: HTTP verbs for different operations
  • Representation: Resources can have multiple formats (JSON, XML)
  • Stateless Communication: No client context stored on server

HTTP Methods in REST

Method Operation Example
GET Retrieve data GET /api/users
POST Create data POST /api/users
PUT Update data (complete) PUT /api/users/42
PATCH Update data (partial) PATCH /api/users/42
DELETE Remove data DELETE /api/users/42

RESTful Resource Naming


// Collection of resources
GET /api/articles          // Get all articles
POST /api/articles         // Create a new article

// Specific resource
GET /api/articles/123      // Get article with ID 123
PUT /api/articles/123      // Update article with ID 123
DELETE /api/articles/123   // Delete article with ID 123

// Nested resources
GET /api/articles/123/comments       // Get all comments for article 123
POST /api/articles/123/comments      // Add a comment to article 123
GET /api/articles/123/comments/456   // Get comment 456 for article 123

Use nouns (not verbs) and plural forms for resource collections.

HTTP Status Codes

Success (2xx)

  • 200: OK (successful request)
  • 201: Created (resource created)
  • 204: No Content (success, no body)

Client Errors (4xx)

  • 400: Bad Request (invalid syntax)
  • 401: Unauthorized (authentication)
  • 403: Forbidden (permissions)
  • 404: Not Found (resource doesn't exist)
  • 409: Conflict (resource state conflict)
  • 429: Too Many Requests (rate limit)

Redirection (3xx)

  • 301: Moved Permanently
  • 302: Found (temporary redirect)
  • 304: Not Modified (cached content)

Server Errors (5xx)

  • 500: Internal Server Error
  • 502: Bad Gateway
  • 503: Service Unavailable
  • 504: Gateway Timeout

Always return appropriate HTTP status codes with your API responses!

JSON Web Tokens (JWT)

JWTs are self-contained tokens that can securely transmit information between parties. 

// JWT structure (three parts separated by dots)
xxxxx.yyyyy.zzzzz

// Header: Algorithm & token type
{
  "alg": "HS256",
  "typ": "JWT"
}

// Payload: Claims (data)
{
  "sub": "1234567890",
  "name": "John Doe",
  "iat": 1516239022,
  "exp": 1516242622
}

// Signature: Verifies integrity
HMACSHA256(base64UrlEncode(header) + "." + base64UrlEncode(payload), secret)

Using JWT Authentication


// Go Example with Gin & JWT
package main

import (
	"github.com/gin-gonic/gin"
	"github.com/golang-jwt/jwt/v4"
	"net/http"
	"strings"
	"time"
)

// Login route - creates token
func main() {
	r := gin.Default()
	
	r.POST("/api/login", func(c *gin.Context) {
		// Authentication simplified
		user := map[string]interface{}{"id": 123, "username": "john"}
		
		// Create token
		token := jwt.NewWithClaims(jwt.SigningMethodHS256, jwt.MapClaims{
			"userId":   user["id"],
			"username": user["username"],
			"exp":      time.Now().Add(time.Hour).Unix(),
		})
		
		tokenString, _ := token.SignedString([]byte("your-secret-key"))
		c.JSON(http.StatusOK, gin.H{"token": tokenString})
	})
	
	r.GET("/api/protected", authMiddleware(), func(c *gin.Context) {
		user, _ := c.Get("user")
		c.JSON(http.StatusOK, gin.H{"message": "Protected data", "user": user})
	})
	
	r.Run(":8080")
}

// Middleware to verify token
func authMiddleware() gin.HandlerFunc {
	return func(c *gin.Context) {
		auth := c.GetHeader("Authorization")
		if auth == "" || !strings.HasPrefix(auth, "Bearer ") {
			c.AbortWithStatus(http.StatusUnauthorized)
			return
		}
		
		tokenString := strings.TrimPrefix(auth, "Bearer ")
		claims := jwt.MapClaims{}
		
		token, err := jwt.ParseWithClaims(tokenString, claims, func(t *jwt.Token) (interface{}, error) {
			return []byte("your-secret-key"), nil
		})
		
		if err != nil || !token.Valid {
			c.AbortWithStatus(http.StatusForbidden)
			return
		}
		
		c.Set("user", claims)
		c.Next()
	}
}

API Error Handling

Proper error handling improves API usability and debugging.

// Standard error response format
{
	"error": {
	"status": 400,
	"code": "VALIDATION_ERROR",
	"message": "Invalid input data",
	"details": [
		{
		"field": "email",
		"message": "Must be a valid email address"
		}
	]
	}
}

// Go/Gin error handling middleware
package main

import "github.com/gin-gonic/gin"

// AppError is our custom error type
type AppError struct {
	Status  int         `json:"status"`
	Code    string      `json:"code"`
	Message string      `json:"message"`
	Details interface{} `json:"details,omitempty"`
}

// Error middleware for centralized error handling
func ErrorMiddleware() gin.HandlerFunc {
	return func(c *gin.Context) {
		c.Next() // Process request
		
		// Check for errors
		if len(c.Errors) > 0 {
			err := c.Errors.Last().Err
			
			// Default to 500 internal error
			status := http.StatusInternalServerError
			response := AppError{
				Status:  status,
				Code:    "SERVER_ERROR",
				Message: "Internal Server Error",
			}
			
			// Use custom error details if available
			if appErr, ok := err.(*AppError); ok {
				response = *appErr
				status = appErr.Status
			}
			
			c.JSON(status, gin.H{"error": response})
		}
	}
}

Input Validation

Always validate all incoming data to ensure integrity and security.

// Go/Gin validation with binding tags
package main

import (
	"github.com/gin-gonic/gin"
	"net/http"
)

// User with validation rules as struct tags
type User struct {
	Username  string `json:"username" binding:"required,alphanum,min=3,max=30"`
	Email     string `json:"email" binding:"required,email"`
	Password  string `json:"password" binding:"required,min=8"`
	BirthYear int    `json:"birthYear" binding:"required,min=1900,max=2025"`
}

func main() {
	r := gin.Default()
	
	r.POST("/api/users", func(c *gin.Context) {
		var user User
		
		// Validate request body
		if err := c.ShouldBindJSON(&user); err != nil {
			// Convert validation errors to our format
			validationErrors := []map[string]string{}
			
			// In a real app, process error details here
			
			c.JSON(http.StatusBadRequest, gin.H{
				"error": gin.H{
					"status":  400,
					"code":    "VALIDATION_ERROR",
					"message": "Invalid input data",
					"details": validationErrors,
				},
			})
			return
		}
		
		// Process valid user data
		c.JSON(http.StatusCreated, gin.H{"message": "User created successfully"})
	})
	
	r.Run(":8080")
}

API Versioning

Versioning allows you to evolve your API without breaking clients.

URL Versioning

https://api.example.com/v1/users
https://api.example.com/v2/users

Pros: Explicit, clear cache separation

Cons: URL changes, harder to maintain

Header Versioning

Accept: application/vnd.example.v1+json
Accept: application/vnd.example.v2+json

Pros: Same URL, cleaner API

Cons: Less discoverable, harder to test

Best Practice: URL versioning is most common and easiest for consumers to understand.
